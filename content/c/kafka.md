# kafka

A distributed event store and stream-processing platform.

## Architecture

Kafka **clusters** are composed of **brokers**, storing **topic partitions**.

### Brokers

Brokers are Kafka server processes. They host partion(s) of topic(s). The file
structure is roughly:

```txt
/var/lib/kafka-logs/
  topicA-0/
    00000000000000000000.log
    00000000000000000000.index
    00000000000000000000.timeindex
```

* `topicA-0`: Topic `A`, Partition `0`.
* `.log`: data
* `.index`: offset (file position info)
* `.timeindex`: a timestamp that can correlate to a log's offset

### Topics and Partitions

Topics are logical namespaces for data. Typically topics represent one "type"
of data, but Kafka does not do schema enforcement. Topics contain one or more
partitions. Partitions divide topics up into multiple logs. Broker leadership
is done at the partition level, which means within a topic, there may be
different brokers leading each parition.

Partitioning of topics enables multiple features, such as:

* Parallelism: consumers (in a group) can each get assigned unique paritions.

* Load distribution: In clusters, a broker may be a leader (handles writes and
  (potentially all) reads) for one topic parition, yet another broker may be
  the leader for another parition, in the same topic.

## APIs

Kafka's primary APIs include:

* **Producer/Consumer**: direct Kafka read/write operations.
* **Connect**: stream data to (sink) or from (source) an external system.
* **Streams**: Continues computations over streams with built in capabilities like
  filtering and grouping. Abstraction built atop Producer/Consumer.

## Kafka Connect

Kafka connect is often used to take a datasource and make it available as a
stream of change. This stream of change can then be used by clients or other
Kafka connect sinks to transfer to other target systems. The process of
streaming change is referred to as Change Data Capture (CDC).
